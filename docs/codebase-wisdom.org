* In Literate Org, Naked Lisp After `#+end_src` Is Just Text
  Date: 2026-04-17

** The Bug
An Astro section in [[file+emacs:/Users/jay/emacs/emacs-settings/shared-functions.org][shared-functions.org]] contained real-looking Emacs Lisp after a =#+end_src=. That made the section look like live config even though the code never tangled into [[file+emacs:/Users/jay/emacs/emacs-settings/shared-functions.el][shared-functions.el]].

** The Truth
- Example blocks are intentional documentation and are fine.
- The real gotcha is naked Lisp-looking text outside source blocks: it is easy to mistake for active config when reading the Org file.
- The safest test is the tangled file, not the Org prose. If the code is absent from [[file+emacs:/Users/jay/emacs/emacs-settings/shared-functions.el][shared-functions.el]], it is not live.

** The Fix
Either move intended live code back into a =#+begin_src emacs-lisp= block or delete stale text if the feature is no longer wanted.

** Lessons Learned / Prevention Checklist
- Do not treat =#+begin_example= as a bug by itself; that is often intentional documentation.
- When an Org section looks suspicious, compare it directly against the tangled =.el= file before changing behavior.
- If a setup is intentionally not used anymore, delete stale pseudo-config instead of leaving it in ambiguous plain text.

  Date: 2026-04-17

** The Bug
Updating packages in Spacemacs failed inside Quelpa while cloning =helm-make= from:
=https://github.com/myrgy/helm-make.git= on branch =add_emacs_completion=.

At first glance this looked like it might be caused by local config in [[file+emacs:/Users/jay/emacs/emacs-settings/shared-functions.org][shared-functions.org]], because that file carries a lot of custom package setup.

** The Truth
- The live =helm-make= recipe does not come from [[file+emacs:/Users/jay/emacs/emacs-settings/shared-functions.org][shared-functions.org]].
- It comes from Spacemacs’s =compleseus= layer at [[file+emacs:/Users/jay/emacs/spacemacs/layers/+completion/compleseus/packages.el][/Users/jay/emacs/spacemacs/layers/+completion/compleseus/packages.el]].
- That layer uses a forked repo because upstream =abo-abo/helm-make= still lacks the =emacs= completion choice needed by this config’s =helm-make-completion-method 'emacs=.
- The fork and branch still exist, so a clone error here is not automatically evidence of a dead upstream; it can also be a transient Quelpa/build-dir failure.

** The Fix
Freeze =helm-make= in [[file+emacs:/Users/jay/emacs/emacs-settings/spacemacs.d/init.el][/Users/jay/emacs/emacs-settings/spacemacs.d/init.el]] via =dotspacemacs-frozen-packages= so bulk package updates do not keep re-entering that fragile forked-recipe path.

** Lessons Learned / Prevention Checklist
- When a Spacemacs package update fails, search both your personal config and the relevant Spacemacs layer before assuming the recipe is yours.
- If a package depends on a fork-only Quelpa recipe and is already working locally, freezing it can be safer than patching vendor code or chasing intermittent update failures.
- Before changing a recipe, verify whether the fork exists and whether the exact =git clone= command works outside Emacs.

* Old `use-package` + VC Installs: `:vc` May Parse Fine In Theory And Still Do Nothing In Practice
  Date: 2026-04-06

** The Bug
Adding
= (use-package org-mindmap :vc (...)) =
to [[file+emacs:/Users/jay/emacs/emacs-settings/shared-functions.org][shared-functions.org]] looked reasonable on Emacs 29, but the package never actually installed or loaded in the running Spacemacs session.

** The Truth
- Emacs 29 has =package-vc-install=, but this setup still loads an older ELPA =use-package= build from =/Users/jay/.emacs.d/elpa/use-package-20210207.1926/=.
- That build does not recognize the =:vc= keyword, so relying on inline VC installation inside =use-package= is brittle here.
- Even after installing =org-mindmap= with =package-vc-install=, the package was not automatically visible to the running session through the normal package activation path.
- The reliable signal came from the live server, not batch bootstrap:
  - =emacsclient -e '(locate-library "org-mindmap")'= was initially =nil=
  - evaluating the exact =use-package= form with an explicit =:load-path= made =(featurep 'org-mindmap)= return =t=

** The Fix
Use the repo's existing VC-package pattern and point the package at the installed checkout explicitly:
#+begin_src elisp
;; /Users/jay/emacs/emacs-settings/spacemacs.d/init.el
(org-mindmap :vc-backend Git :url "https://github.com/krvkir/org-mindmap.git")

;; /Users/jay/emacs/emacs-settings/shared-functions.org
(use-package org-mindmap
  :ensure nil
  :load-path "/Users/jay/.emacs.d/elpa/org-mindmap"
  :after org)
#+end_src

** Lessons Learned / Prevention Checklist
- Do not assume Emacs 29 implies modern =use-package= keyword support; check the actual installed =use-package= version.
- If this config already has a package-vc convention, follow it instead of mixing in inline =:vc= declarations.
- For packages that are present on disk but invisible at runtime, test in the live server with =emacsclient= before trusting batch bootstrap output.
- When package activation is flaky, an explicit =:load-path= to the VC checkout can be the least risky fix.

* Transcript Cleanup Regex: `[[:space:]]*$` Can Destroy Org Structure
  Date: 2026-04-01

** The Bug
Running =whittle-transcript= on a diarized Org transcript could remove the blank line before the next paragraph or =*** Speaker:= heading.

** The Truth
- In =whittle/transcript-edge-filler-rules=, the sentence-final filler rule matched trailing =okay/right/you know= with =[[:space:]]*$=.
- In Emacs regexps, =[[:space:]]= includes newlines, not just spaces and tabs.
- That meant cleaning a line like =Is everything okay?= could also consume the following paragraph break, collapsing transcript structure even though the command was only meant to remove a filler phrase.
- The same rule family also lacked word boundaries, so =so= at sentence start could match the first two letters of a real word like =Sometimes=.

** The Fix
Use =[[:blank:]]*= in end-of-line cleanup rules that should preserve layout, do not classify sentence-final =okay= as filler, and require word boundaries around filler tokens:
#+begin_src elisp
"^[[:space:]]*\\(?:\\<so\\>\\|\\<well\\>\\|\\<ok\\(?:ay\\)?\\>\\|\\<right\\>\\|\\<you know\\>\\|\\<like\\>\\)[,[:space:]-]*"
"[[:blank:]]*,?[[:blank:]]*\\(?:\\<right\\>\\|\\<you know\\>\\)\\([.?!]?\\)[[:blank:]]*$"
#+end_src

** Lessons Learned / Prevention Checklist
- In cleanup regexps, treat =[[:space:]]= as "may eat newlines" unless you explicitly want multiline behavior.
- For prose/transcript normalization, prefer =[[:blank:]]= when trimming inline spacing near punctuation or end-of-line fillers.
- Be especially conservative with common content words like =okay=; they often carry real semantic meaning in transcripts.
- Add word boundaries around short filler tokens like =so= and =like=, or they can silently damage legitimate words.
- If a text-cleanup command mysteriously collapses Org headings or paragraph breaks, audit any regex that ends with =$= and includes =[[:space:]]=.

* Org Export Hooks: Avoid `replace-regexp` In Before-Parsing Hooks
  Date: 2026-03-14

** The Bug
Astro export crashed with:
=wrong-type-argument number-or-marker-p nil=
inside =isearch-clean-overlays= / Magit advice, even though the exporter itself was fine.

** The Truth
- A global =org-export-before-parsing-functions= hook in =shared-functions.org= used buffer-level =replace-regexp= to emphasize leading =Label:= prefixes.
- =replace-regexp= runs interactive replacement cleanup (=replace-dehighlight=), which calls =isearch-clean-overlays=.
- If another package leaves behind a stale overlay, Magit's overlay cleanup advice can fail during export.
- Anonymous export-hook lambdas are also harder to inspect, remove, or selectively sanitize later.

** The Fix
Replace the anonymous hook with a named function that edits the buffer directly via =re-search-forward= and =replace-match=:
#+begin_src elisp
(defun jay/org-export-emphasize-leading-labels (_backend)
  "Emphasize leading \"Label:\" prefixes without touching URL schemes."
  (save-excursion
    (goto-char (point-min))
    (while (re-search-forward "^\\([A-Za-z]+:\\)\\([^/]\\|/[^/]\\|$\\)" nil t)
      (replace-match "*\\1*\\2" nil nil))))
#+end_src

** Lessons Learned / Prevention Checklist
- Avoid =replace-regexp= / =query-replace= inside export hooks when a plain scan-and-replace will do.
- Prefer named functions over anonymous lambdas in hooks so they can be debugged and removed cleanly.
- If an export crash looks like an exporter bug but the stack mentions =replace-regexp=, inspect global export hooks first.
- Gate hook behavior by backend when appropriate; global export hooks leak into every exporter, including custom ones like =ox-astro=.

* Org 9.8 Preview Migration: Guard For Mixed Runtime Versions
  Date: 2026-02-24

** The Bug
We migrated to the new Org 9.8 preview API (=org-link-preview= and link =:preview=), but the current runtime was still Org 9.6.15 in batch checks. New code loaded, but 9.8-only entry points were unavailable.

** The Truth
- API migration and runtime upgrade are separate events.
- If config calls 9.8 functions unconditionally while Org is still 9.6/9.7, behavior regresses or errors at call sites.
- Custom link preview integrations (like =org-yt=) need both paths during transition.

** The Fix
- Added feature guards using =fboundp=.
- Introduced a compatibility wrapper that prefers =org-link-preview= when available, else falls back to =org-display-inline-images= (and then =org-toggle-inline-images= for very old setups).
- For =org-yt=, kept the 9.8 =:preview= parameter and added a legacy advice fallback on =org-display-inline-images= when =org-link-preview= is missing.

** Lessons Learned / Prevention Checklist
- Before adopting a new Org API, verify runtime with batch:
  - =emacs --batch --eval "(progn (require 'org) (princ org-version))"=
  - =emacs --batch --eval "(progn (require 'org) (princ (fboundp 'org-link-preview)))"=
- During migrations, write compatibility shims first, then switch call sites to the shim.
- Prefer symbol guards (=fboundp= / =boundp=) over version-string comparisons.
- After literate edits, keep the loop tight: edit → tangle → =check-parens= → smoke test.

* Org Src Blocks: Headings Inside Break Tangling
  Date: 2026-01-27

** The Bug
=endless/upcase= and related functions were autoloaded from =spacecraft-mode.el= but the =.el= didn't contain them. Calling =M-u= produced: =Autoloading file ... failed to define endless/upcase=.

** The Truth
A top-level =* heading= inside a =#+begin_src ... #+end_src= block silently breaks org's block parser. Everything after the heading is no longer considered part of the block, so it doesn't tangle. No error is raised — the functions simply vanish from the output.

** Why This Was Tricky
- The =.org= file looked correct at a glance — the functions were between =#+begin_src= and =#+end_src=
- =org-babel-tangle= in batch mode reported "Tangled 0 code blocks" with no error
- The =.el= file existed (from a previous tangle or manual creation) so nothing seemed missing until the autoload fired

** The Fix
Move =#+end_src= to before the =* heading= so the block closes properly:
#+begin_src org
)
)

,#+end_src

,* spacecraft-mode - heading goes here

,** smart-space
#+end_src

** Lessons Learned / Prevention Checklist
- Never put an org heading (=*=) inside a src block — it silently breaks tangling
- When =org-babel-tangle= reports 0 blocks, check for structural issues (mismatched or interrupted blocks), not just missing =:tangle= headers
- After editing literate org files, always verify the tangled =.el= contains the expected =defun= forms
- When autoloads fail, check that the target =.el= actually defines the function — don't just trust the autoload declaration

* Global Minor Mode Hooks: Re-enable After Minibuffer
  Date: 2026-01-27

** The Bug
All =key-minor-mode-map= bindings (=s-a=, =s-c=, =s-k=, =s-<up>=, etc.) worked at startup but became undefined after the first minibuffer interaction.

** The Truth
=key-minor-mode= is =:global t=. The =minibuffer-setup-hook= called =(key-minor-mode -1)= to disable it in the minibuffer, but since it's a global mode, this disabled it everywhere — and nothing re-enabled it.

** The Fix
#+begin_src elisp
(add-hook 'minibuffer-exit-hook (lambda () (key-minor-mode 1)))
#+end_src

** Lessons Learned / Prevention Checklist
- Disabling a =:global t= minor mode affects all buffers, not just the current one
- Always pair a =setup-hook= disable with an =exit-hook= re-enable
- If bindings "work at startup but die later," suspect a hook that disables a global mode

* Prefix Maps: fset Clobbers Populated Keymaps
  Date: 2026-01-27

** The Bug
All =s-k= sequences (=s-k s-k=, =s-k g l=, etc.) were undefined after restart.

** The Truth
=keys.el= defined =s-k-prefix= via =define-prefix-command= and populated it with all =s-k ...= bindings. Then =gnu-emacs-startup.el= ran =(fset 's-k-prefix s-k-map)= inside =(with-eval-after-load 'evil ...)=, replacing the populated keymap with a brand new empty one.

** The Fix
Removed the =fset= block from =gnu-emacs-startup.el=; =keys.el= now solely owns the prefix definition.

** Lessons Learned / Prevention Checklist
- =fset= on a prefix command symbol replaces the entire keymap — any bindings installed before the =fset= are lost
- When consolidating keybindings into one file, grep for =fset=, =defvar=, and =make-keymap= on the same symbol in other files
- =with-eval-after-load= runs later than top-level code, so it can silently clobber earlier setup

* Org Emphasis Regex: Must Rebuild After Changing Components
  Date: 2025-12-26

** The Bug
Org emphasis (~*bold*~, ~/italic/~) stopped being fontified in some buffers. Inline emphasis like ~this *word* here~ wouldn't render, though emphasis on its own line sometimes worked.

** The Truth
Setting ~org-emphasis-regexp-components~ alone does nothing—org caches the compiled regex at load time. The variable is just data; ~org-set-emph-re~ must be called to rebuild ~org-emph-re~ from it.

** Why This Was Tricky
- Evaluating the setq interactively followed by ~org-set-emph-re~ worked fine, making it seem like the config was correct
- The cached regex persisted across buffer switches, so the problem appeared file-specific
- Restarting Emacs loaded the setq but never rebuilt the regex, so emphasis silently failed

** The Fix
Call ~org-set-emph-re~ after org loads:
#+begin_src elisp
(setq org-emphasis-regexp-components
      '("-[:space:]('\"{" "-[:space:].,:!?;'\")}\\[" "[:space:]" "." 1))
(with-eval-after-load 'org
  (org-set-emph-re 'org-emphasis-regexp-components org-emphasis-regexp-components))
#+end_src

** Lessons Learned / Prevention Checklist
- Org caches expensive regexes at load time; changing the source variable requires an explicit rebuild call
- When a setq "works interactively but not on restart," suspect a missing initialization hook
- Always pair ~org-emphasis-regexp-components~ changes with ~org-set-emph-re~
- For emphasis issues, check ~org-emph-re~ value—if it doesn't match your components, the rebuild never happened

- When rewriting lists, gather matches and mutate afterwards; in-place loop edits are fragile on long AI outputs.

* Org Heading Flattening: Structure vs Formatting
  Date: 2025-10-13

** The Bug
Pasting an Org subtree via `pasteboard-paste-adaptive` turned every `**** TODO` entry into `*** TODO`, collapsing the heading hierarchy in-place.

** The Truth
The cleaning pass ran `replace-smart-quotes`, which still contained the Markdown-era rule “replace every `**` with `*`”. During the “clean” paste path we already had Org syntax, so that blanket replacement ate half of each heading’s star run before the text ever touched the buffer.

** Why This Was Tricky
- The regression only appeared in Org buffers that took the “clean” branch; Markdown/code buffers went through the verbatim branch and stayed fine.
- Unit snippets focused on Markdown conversions, so pasting native Org trees wasn’t part of the smoke tests.
- Watching the final buffer hid the culprit; the destructive change happened inside the temp buffer that feeds `pasteboard-paste`.

** The Fix
Teach the `** → *` loop to walk the line’s leading stars, record the first non-star position, and only convert `**` pairs that appear *after* that point. Tangled the change so both `.org` and `.el` stay in sync.

** Lessons Learned / Prevention Checklist
- Distinguish between structure (`^\\*+ ` headings) and inline formatting (`*bold*`) before reusing Markdown cleanup on Org text.
- When editing temp buffers, add `(message ...)` or batch snippets to observe the intermediate state—otherwise you miss where the hierarchy was flattened.
- Expand paste fixtures to include “pure Org” inputs with nested headings, inline bold, and emojis; they exercise the same sanitizers as Markdown pastes.
- Any rule that blindly rewrites punctuation (`**`, `--`, etc.) must document the assumed input syntax and bail out when that assumption no longer holds.

** Follow-up: Org-Fold + `replace-regexp`
- Symptom: `pasteboard-paste-adaptive` raised “Calling `org-fold-core-region` with missing SPEC” while `replace-smart-quotes-regexp` ran in a temp buffer.
- Cause: `replace-regexp` triggers Org’s fold maintenance, which expects the interactive search state; our non-interactive call provided no fold spec.
- Fix: Replaced the `mapcar` call to `replace-regexp` with manual `re-search-forward`/`replace-match` loops so Org never tries to manage folds in the temp buffer.
- Guardrail: Avoid `replace-regexp` (and other interactive wrappers) inside hidden buffers; prefer explicit loops that stay fold-agnostic.

* Sentence Boundary Probes Can Roll Back Atomic Changes
  Date: 2025-10-06

** The Bug
`my/delete-backward-and-capitalize` appeared to do nothing when run on a selected region—especially `mark-whole-buffer` followed by the command.

** The Truth
Inside the command, `my/beginning-of-sentence-p` calls `forward-sentence` to detect sentence starts. When the region spans the whole buffer, the delete happens first, leaving an empty buffer. `forward-sentence` then signals `end-of-buffer`, and because we are inside an `atomic-change-group`, Emacs rolls back the entire transaction, silently restoring the buffer.

** The Fix
Wrap the sentence probe in `ignore-errors` and guard against a nil return so we bail out gracefully instead of throwing.

** Lessons Learned / Prevention Checklist
- Treat navigation helpers invoked inside `atomic-change-group` or `combine-after-change-calls` as potential rollback triggers; guard them against `error`.
- When a command "does nothing," rerun with `toggle-debug-on-error`—silent rollbacks usually come from trapped errors.
- Add fixtures that cover empty-buffer or whole-buffer deletions whenever a helper mixes editing and sentence/paragraph navigation.

* Region Boundary Invalidation: Capture After Modification Check
  Date: 2025-10-03

** The Bug
When selecting all text in a buffer and calling `my/delete-backward-and-capitalize`, the function would fail to delete the highlighted region correctly.

** The Truth
During cache stabilization refactoring, region boundaries were captured in a `let*` binding BEFORE checking if buffer modifications (like heading cleanup) would run. The heading cleanup (`kill-line 0`) invalidated the captured positions before they were used for deletion.

** Why This Happened
The refactor moved from a messy but working version (where heading check was at the top, outside main logic) to a cleaner `let*` binding structure. The developer ordered the bindings logically but didn't account for the critical timing: positions must be captured AFTER determining whether intervening modifications will occur, or modifications must be prevented when positions are needed.

** The Fix
Changed from:
#+BEGIN_SRC emacs-lisp
(let* ((at-heading (looking-back-safe "^[*]+ "))
       (has-region (use-region-p))
       (region-beg (when has-region (region-beginning)))
       (region-end (when has-region (region-end)))
       ...)
#+END_SRC

To:
#+BEGIN_SRC emacs-lisp
(let* ((has-region (use-region-p))
       (at-heading (and (not has-region) (looking-back-safe "^[*]+ ")))
       (region-beg (when has-region (region-beginning)))
       (region-end (when has-region (region-end)))
       ...)
#+END_SRC

Now `at-heading` is only true when there's no region, preventing buffer modifications before region deletion.

** Lessons Learned / Prevention Checklist
- When refactoring for cleanliness, watch for position captures that depend on buffer state
- In `let*` bindings that capture positions: either (1) capture AFTER all checks that might modify buffer, or (2) guard modifications to prevent them when positions are needed
- Buffer positions are invalidated by ANY text modification between capture and use - even "harmless" cleanup
- Test region-based operations during refactoring, especially when reordering logic
- The same pattern could affect other functions: check for `(region-beginning)` or `(region-end)` captured before buffer modifications

* Markdown Bold Conversion: Order and Precision Matter
  Date: 2025-10-05

** The Bug
When pasting markdown text like `## **Dates:** October 7-9`, the output was incorrectly formatted as `** **Dates:**` or `***Dates:*` instead of the desired `** *Dates:*`.

** The Truth
The issue had TWO layers:

1. *Order of operations bug*: Originally, `replace-smart-quotes` ran BEFORE `convert-markdown-headings-to-org`:
   - Input: `## **Dates:**`
   - `replace-smart-quotes` converted `**` → `*`: `## *Dates:*`
   - `convert-markdown-headings-to-org` converted `##` → `**`: `** *Dates:*`
   - But this left the `##` unconverted in some cases

2. *Heading detection bug*: The `**` → `*` replacement logic checked if the LINE started with asterisks, but then skipped ALL `**` on that line:
   - Input (after heading conversion): `** **Dates:**`
   - Check: "Does line start with `^\*+\s`?" → Yes (matches `** `)
   - Action: Skip ALL `**` on the line (even the ones in `**Dates:**`)
   - Result: `** **Dates:**` (nothing replaced)

** The Fix
1. *Fixed order*: Run `convert-markdown-headings-to-org` FIRST, before any asterisk processing in `pasteboard--clean-string`
2. *Fixed heading detection*: Only skip `**` if it's PART OF the heading stars at line start:
   #+begin_src elisp
   (let* ((match-pos (match-beginning 0))
          (line-start (line-beginning-position))
          (at-line-start (= match-pos line-start)))
     ;; Only skip if THIS specific ** is at line start AND part of heading
     (if (and at-line-start (looking-at "^\\*+\\s-"))
         nil  ; Skip - it's heading stars
       (replace-match "*" t t)))  ; Replace - it's markdown bold
   #+end_src

** Lessons Learned / Prevention Checklist
- When processing mixed markdown/org content, ORDER MATTERS: Convert structural elements (headings) FIRST, then process inline formatting (bold, italic)
- Always check the SPECIFIC match location, not just the line pattern
- Test with complex cases like `## **Bold:** text` that combine structural and inline markdown
- Document cleaner order in architecture docs and enforce it with comments in code

* Org Heading Regex: Always Require Space After Stars
  Date: 2025-10-10

** The Bug
Bold text at the start of lines (`*"text"*`) was being treated as org headings by functions that match `^\\*+`, causing incorrect transformations.

** The Truth
Org headings ALWAYS have a space after the stars (`** Heading`), but many regex patterns forget this distinction and match ANY line starting with asterisks, including bold/emphasis markers at line start.

** The Fix
Change patterns from `"^\\(\\*+\\)"` to `"^\\(\\*+\\) "` when matching actual org headings. Include the space in replacements too:
#+BEGIN_SRC emacs-lisp
;; Wrong - matches bold text at line start
(re-search-forward "^\\(\\*+\\)" nil t)

;; Right - only matches org headings
(re-search-forward "^\\(\\*+\\) " nil t)
(replace-match (concat new-stars " ") t t)  ; Keep the space
#+END_SRC

** Lessons Learned / Prevention Checklist
- Org structural elements (headings) vs inline formatting (bold) have distinct patterns
- A heading is `^\\*+ ` (stars + space), bold is just `*text*`
- When processing org text after markdown conversion, remember you're seeing org syntax not markdown
- Test with bold/italic text at line boundaries - common edge case for pattern bugs
- Document whether functions expect markdown input or org input

* Emoji in LaTeX Export: Filter at the Right Level
  Date: 2025-10-05

** The Bug
Emoji like 📅 🎯 💡 were appearing in LaTeX output, causing compilation errors since standard LaTeX doesn't support emoji.

** The Truth
The initial fix only added an `org-export-filter-plain-text-functions` filter, but emoji in headings don't pass through the plain-text filter - they're part of the headline structure.

#+begin_src elisp
(defun my-org-latex-strip-emoji (text backend info)

    (replace-regexp-in-string "[🀀-🫿]" "" text)))

(add-to-list 'org-export-filter-plain-text-functions 'my-org-latex-strip-emoji)
(add-to-list 'org-export-filter-headline-functions 'my-org-latex-strip-emoji)
#+end_src

- Org export has different filter types for different content types (headlines, paragraphs, links, etc.)
