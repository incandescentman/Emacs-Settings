#+TITLE: Startup Optimization Plan
#+DATE: 2026-04-18
#+DESCRIPTION: Measure-first plan for reducing Emacs startup time without cutting needed functionality

* Goal

Reduce Emacs startup time only by removing unnecessary startup work.

Do not cut features just because they are expensive.
Do not assume that the largest file is the most disposable.
Prefer deferring or restructuring important code over deleting it.

* Working Principles

1. Measure first, then optimize.
2. Preserve important behavior unless there is clear evidence it is unused.
3. Prefer delaying interactive helpers until first use.
4. Prefer changing how code is loaded before deleting code.
5. Treat large literate files as structural hotspots, not automatic deletion targets.

* Phase 3: Choose the Least Disruptive Optimization

Only pick from the options below after the audits.

** Option 1: Defer unnecessary startup work

Use when:
- code is genuinely only needed after the first command invocation

Examples:
- search helpers
- optional interactive packages
- non-essential mode setup

This is the safest kind of performance improvement.

** Option 2: Keep features, change load strategy

Use when:
- the code is important
- the file is expensive because of literate loading or eager startup structure

Examples:
- keep the same functionality but avoid paying startup =org-babel-load-file= cost for every launch
- load tangled =.el= files directly when appropriate
- move expensive setup behind idle timers or command entry points

This is the likely path for [[file+emacs:/Users/jay/emacs/emacs-settings/shared-functions.org][/Users/jay/emacs/emacs-settings/shared-functions.org]] if the functionality is still needed.

** Option 3: Remove dead code

Use only when:
- the code is clearly unused
- the behavior is obsolete
- there is no workflow loss

This should be the last option, not the default one.

* Decision Rules

Make changes only if one of these is true:

1. The code is loaded at startup but not needed until later.
2. The code is duplicated or obsolete.
3. The expensive part is structural and can be loaded more cheaply without losing behavior.

Do not make changes just because:

1. a file is large
2. a feature is old
3. a setting looks messy
4. a package is not fashionable
